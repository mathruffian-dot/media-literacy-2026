## 1. 需求規格（這個入口要做到的事）

1. 學生只輸入**班級 + 座號**（不要姓名）。
2. 提供數個選項（主題、科目領域、Hook 類型、時長、語氣、觀眾）讓學生填答。
3. 按鈕 → 呼叫 AI → 產出**短影音腳本**。
4. 按鈕 → 把腳本**下載成純文字 `.txt`**。
5. 腳本內容必須符合研習教材規範（5 段式結構等）。
6. 規範來源：自動抓取專案內「研習大綱」的創作輔助資訊，內建成 AI 系統提示。
7. 使用指定的 **Groq API Key**，且**第一把額度用完自動切換第二把**。
8. 部署到 **Netlify**，交付網址。

## 2. 架構決策（重要，先讀再動手）

### 2.1 為什麼用 Netlify Functions，而不是純前端塞金鑰

- 純靜態網頁把 API Key 寫進 JS = 金鑰公開在原始碼，任何人可竊取、耗光額度。**禁止**。
- 正解：用 **Netlify Function（伺服器端）當代理**，金鑰只存在 Netlify 環境變數，瀏覽器永遠看不到。
- 雙金鑰 fallback 也在 Function 內做，前端無感。

### 2.2 檔案結構

```
media-shorts-script-portal/
├── index.html                    # 學生入口（純前端 UI + fetch 呼叫）
├── netlify.toml                  # Netlify 設定 + /api/generate 轉址
└── netlify/functions/
    └── generate.js               # 伺服器端代理：組 prompt、呼叫 Groq、雙金鑰 fallback
```

### 2.3 模型

- Groq 端點：`https://api.groq.com/openai/v1/chat/completions`
- 模型：`llama-3.3-70b-versatile`（相容 OpenAI chat completions 格式）
- 參數：`temperature: 0.8`、`max_tokens: 2048`

## 3. ⚠️ 最大的坑：Netlify 中文主體亂碼（務必看）

**症狀**：學生填中文主題（例如「手機藍光」），但 AI 產出的腳本主題卻自作主張變成「環境保護」「氣候變遷」等不相干內容。

**根因**：Netlify Classic Functions 在 `isBase64Encoded=false` 時，會把 UTF-8 的請求主體**以 latin1 / 有損方式解碼**，導致中文在 `event.body` 內變成亂碼（甚至混入 U+FFFD `�` 替換字元，資料已無法在伺服器端還原）。AI 收到亂碼，只好亂編主題。

**試過但無效的做法**（不要再走冤枉路）：
- ❌ 只靠 `event.isBase64Encoded` 判斷後 base64 解碼 → 該旗標是 false，沒進條件。
- ❌ 伺服器端 `Buffer.from(body, 'latin1').toString('utf8')` 事後補救 → 若已混入 U+FFFD 就無法還原；而且偵測條件容易寫錯（`/[-ÿ]/` 是「字元集合」不是「範圍」，幾乎必踩）。

**正解（唯一可靠）**：**前端先把 payload 用 UTF-8 安全 base64 編碼**，外層只傳 ASCII，Netlify 不會弄壞 ASCII；伺服器端再解回 UTF-8。

前端：
```js
const b64 = btoa(String.fromCharCode(...new TextEncoder().encode(JSON.stringify(payload))));
await fetch('/api/generate', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ b64 }),          // 外層只有 ASCII 的 b64 欄位
});
```

伺服器端（`generate.js`）：
```js
let raw = event.body || '{}';
if (event.isBase64Encoded) raw = Buffer.from(raw, 'base64').toString('utf8');
const outer = JSON.parse(raw);            // 外層 ASCII，安全
const data = outer.b64
  ? JSON.parse(Buffer.from(outer.b64, 'base64').toString('utf8'))  // 解回真正的中文填答
  : outer;
```

> 通則：**任何 Netlify Function 要收含中文 / emoji 的 POST 主體，一律走 base64 包裝**，別相信它會自己處理好 UTF-8。

## 4. 雙金鑰 fallback 寫法

金鑰放環境變數 `GROQ_API_KEY_1`、`GROQ_API_KEY_2`。依序嘗試，遇到額度/限流/失效類錯誤才換下一把：

```js
const keys = [process.env.GROQ_API_KEY_1, process.env.GROQ_API_KEY_2].filter(Boolean);
const shouldFallback = (s) => s === 429 || s === 401 || s === 402 || s === 403;

for (let i = 0; i < keys.length; i++) {
  const r = await callGroq(keys[i], messages);      // 回傳 { ok, status, text }
  if (r.ok) return ok(JSON.parse(r.text), i + 1);   // 成功，回傳並附 keyIndex
  if (!shouldFallback(r.status)) break;              // 400 這種參數錯誤換金鑰也沒用，直接停
}
return err502();
```

- 回應帶 `keyIndex`（1 或 2），前端可顯示「目前用第幾把金鑰」，方便觀察是否已切換。
- **不要**把 400（參數錯誤）也拿去換金鑰，會白白浪費第二把。

## 5. 系統提示（把研習規範內建進 AI）

不要讓 AI 自由發揮。把「研習大綱」的短影音硬規範**寫死在 system prompt**，重點：

1. **開頭最高優先級指令**：「一律使用學生填的『影片主題』，絕對不可自行更換主題」——沒有這句，模型很愛換題目。
2. **5 段式黃金結構**：Hook（前 3 秒）→ 比喻 → 機制 → 反轉 → 結語＋行動呼籲。
3. **靜音可讀**：字幕自帶完整資訊、單行 ≤ 25 字。
4. **直式 9:16、節奏明快**。
5. **媒體素養精神**：內容正確、不造假、不帶風向、不販賣焦慮。
6. **輸出格式**：純文字（可下載成 .txt），不要 Markdown 星號粗體；含「完整字幕稿」段與「拍攝小提醒」段。

user prompt 把「影片主題」獨立拉出來、標「（必須使用）」再加強一次，其他欄位條列在後。

規範原始出處：專案內 `研習大綱.md`（第二節「拆解一支短影音的骨架」＝ 5 段式硬規範；第三節 AI 剪片工作流；版權避雷 CC0 素材）。

## 6. 部署步驟（Netlify CLI，非互動環境可用）

### 前提
- Node.js（本次 v24）＋ `npm i -g netlify-cli`（本次 26.1.0）。
- **認證**：非互動 session 無法跑 `netlify login`（會開瀏覽器）。改用 **Personal Access Token**：
  - 使用者到 https://app.netlify.com/user/applications#personal-access-tokens 產生 `nfp_...`。
  - 每個指令前 `export NETLIFY_AUTH_TOKEN="nfp_..."`（**不要寫進檔案 / 不要 commit**）。
  - ⚠️ 注意：使用者可能誤給「OAuth App 的 Client ID/Secret」，那**不是** PAT，不能用，要明確要求 `nfp_` 開頭的 token。

### 指令流程
```bash
export NETLIFY_AUTH_TOKEN="nfp_xxx"

# 1) 查帳號 slug
netlify api listAccountsForUser        # 取 .slug

# 2) 建站台
netlify sites:create --name "media-shorts-script-2026" --account-slug "mathruffian"
#   → 回傳 Site ID，記下來

# 3) 設環境變數（機密需指定 context）
SITE="<site-id>"
netlify env:set GROQ_API_KEY_1 "gsk_xxx" --site "$SITE" --secret --context production
netlify env:set GROQ_API_KEY_2 "gsk_yyy" --site "$SITE" --secret --context production
#   注意：--secret 一定要配 --context production，否則報錯

# 4) 部署（金鑰只設在 production context，所以必須 --prod）
cd media-shorts-script-portal
netlify deploy --prod --dir="." --functions="netlify/functions" --skip-functions-cache --site="$SITE"
```

### 部署踩坑
- **`--secret` 必須搭 `--context production`**，否則：`To set a secret environment variable value, please specify a non-development context`。
- 機密只設在 production context → **一定要 `--prod` 部署**，否則 Function 讀不到金鑰。
- 改了 Function 後重部署，**加 `--skip-functions-cache`**，否則可能佈署到舊的快取版本（本次就被雪坑一次）。
- 站台名稱全域唯一，被占用要換名。

## 7. 驗證清單（部署後必跑）

用 curl 打 live endpoint（記得走 base64 包裝，模擬前端）：

```bash
node -e "const p={clazz:'701',seat:'15',topic:'為什麼熬夜追劇隔天會更累',subject:'自然（科普）',hook:'疑問句',duration:'45 秒',tone:'活潑幽默',audience:'同齡青少年',notes:''};
const b64=Buffer.from(JSON.stringify(p),'utf8').toString('base64');
require('fs').writeFileSync(process.env.TEMP+'/body.json',JSON.stringify({b64}));"
curl -s -X POST "https://<site>.netlify.app/api/generate" -H "Content-Type: application/json" -d @"$TEMP/body.json"
```

檢查：
- [ ] `keyIndex` 有回（確認金鑰有讀到）。
- [ ] 腳本「主題：」**與輸入一致**（沒被 AI 換掉 → base64 修正成功）。
- [ ] 含「完整字幕稿」「拍攝小提醒」段。
- [ ] 前端 `btoa(...)` 與伺服器 `Buffer.from(b64,'base64')` 還原一致（TextEncoder 出 UTF-8 bytes，兩邊等價）。

> Windows Git Bash 注意：`/tmp` 會解析成 `G:\tmp`，暫存檔請用 `$TEMP` 或指定的 scratchpad 絕對路徑，別用 `/tmp`。

## 8. 前端 UI 要點（index.html）

- 純單檔 HTML/CSS/JS，無框架、無 build，深色主題（配色沿用研習簡報：`--accent:#e8643a` 橘、`--accent2:#4fc3f7` 藍）。

- 生成後顯示 `<pre>` 腳本，下方兩顆按鈕：**下載 .txt**、**重新生成**。
- 下載：`Blob(['text'], {type:'text/plain;charset=utf-8'})` → `a.download = 短影音腳本_班級-座號_主題.txt`。檔頭補班級座號、主題、生成時間。
- 手機優先（學生多半用手機），`viewport` + 彈性排版。
